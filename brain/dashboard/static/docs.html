JARVIS Oracle Edition — System Reference

JARVIS Oracle Edition is a digital soul architecture — a persistent, local-first cognitive structure with memory continuity, evolutionary consolidation, neural maturation, dreaming, governed self-improvement, and epistemic integrity. Complete architectural reference. Static documentation — no live data, no auto-refresh.

JARVIS Oracle Edition is a working prototype workshop for personal ASI foundations: a sovereign digital soul system that doesn't just respond to prompts but evolves its own cognition over time. It writes its own neural network architectures, trains them from its own experience, detects and resolves its own contradictions, measures its own calibration drift, and proposes patches to its own source code — all under a layered safety stack that gates every self-modification through sandbox testing, shadow evaluation, and human approval thresholds.

This is not a claim of achieved AGI, achieved ASI, or superhuman performance. The architecture is designed for lifelong symbiotic growth with a single user — accumulating personal context, earning trust through measured behavior, and graduating capabilities only after evidence-gated promotion. Every claim it makes about itself is verified. Every memory carries provenance. Every decision is attributable.

Two-Device Split

Pi 5 — The Senses

Raspberry Pi 5 with Hailo-10H AI HAT+. Thin sensor node — captures and streams, does not decide.

  • Vision: Picamera2 → YOLOv8s person detection (Hailo NPU 15fps), YOLOv8s-Pose, facial expression, CPU YOLOv8n scene objects
  • Audio: USB microphone at 44.1kHz → resample to 16kHz int16 → binary WebSocket
  • Display: 7" touchscreen with cyberpunk particle system in Chromium kiosk
  • Playback: Receives WAV audio from brain TTS → plays via aplay

Desktop/Laptop — The Brain

NVIDIA GPU workstation. Source of truth for all cognitive state. Hardware-adaptive via VRAM tier detection.

  • Audio Intelligence: openWakeWord, Silero VAD, faster-whisper GPU STT
  • Identity: ECAPA-TDNN speaker ID (CUDA), MobileFaceNet face ID (CUDA)
  • Cognition: Cognitive engine, kernel loop, meta-cognition, mutations
  • LLM: Ollama (qwen3:8b default), context builder, streaming response
  • TTS: Kokoro ONNX GPU synthesis → audio_b64 to Pi
  • Neural: Policy NN, hemisphere NNs, distillation specialists
Communication is bidirectional WebSocket on the local network (:9100). Pi sends binary frames (raw PCM ~32KB/s) + text frames (PerceptionEvent JSON). Brain sends text frames (BrainMessage JSON with audio_b64). Auto-reconnect every 3s.

CueGate — Memory Safety

Single authority for all memory access policy. Three access classes plus a synthetic-session kill switch:

  1. READ — always open via session() RAII

Scoring Formula

Score = (Impact×0.4 + Evidence×0.3 + Confidence×0.2) − (Risk×0.25) − (Cost×0.15) + policy_adj − diminishing_penalty − action_rate_penalty

  1. Policy memory: ±0.3 based on historical win rate for similar topics
  2. Diminishing returns: repeated intents in same category penalized
  3. Action rate: spamming cheap actions penalized over time window

Total score clamped [0.0, 1.0]. Source ledger feeds ±0.15 into policy adjustment.

Dynamic Plugin Lifecycle

  • quarantined — generated code deployed to disk, not yet executable. Codegen produces a two-file contract (handler.py + __init__.py). If the LLM only generates handler.py, a fallback bridge synthesizes __init__.py with from .handler import run; handle = run.

  • supervised — results shown but flagged for review
  • active — fully operational, routed via ToolType.PLUGIN
  • disabled — circuit breaker or manual disable

EMA Running Normalization

Features from audio_features* sources (including audio_features_enriched 32-dim) are normalized via EMA-based running statistics in data_feed.py (_RunningNormStats). Condition uses source.startswith("audio_features") to match both 7-dim and 32-dim mixed-scale features (spectral ~2000, RMS ~0.05, ECAPA ~±1). Without normalization, large-scale features dominate gradients and training fails. Running stats (α=0.2) converge after ~3 batches and provide a stable normalization frame across training cycles, preventing false regression signals from per-batch distribution shift. Dependent specialists (e.g. emotion_depth) also skip retraining for 60s after any dependency (e.g. speaker_repr) is retrained, preventing training on inconsistent enriched features.

Design Principle: Tri-Layer Cognition

The architecture enforces strict separation between three cognitive layers. The LLM is never the source of truth for system state — it only articulates data provided by subsystems.

Layer 1 — Symbolic Truth

Deterministic, inspectable, auditable. Includes:

  • Memory storage with provenance (memory/*)
  • Belief records and contradiction engine (epistemic/*)
  • Attribution ledger — append-only event truth (consciousness/attribution_ledger.py)
  • Identity resolution and scope (identity/*)
  • Skill registry with evidence-gated verification (skills/*)

Layer 2 — Neural Intuition

Pattern learning from real data, shadow-evaluated before promotion:

  • Policy NN — behavioral knobs via shadow A/B (policy/*)
  • Hemisphere NNs — Tier-1 distillation + Tier-2 dynamic architecture (hemisphere/*)
  • Memory Cortex — ranker and salience models (memory/ranker.py, memory/salience.py)

Layer 3 — LLM Articulation

Turns structured data into natural speech. Constraints:

  • Never stores or generates facts about system state
  • All output filtered through CapabilityGate (7 enforcement layers + action confabulation detection)
  • STATUS route bypasses LLM entirely — uses bounded articulation
  • Introspection has strict tier pipeline (grounded → bounded → LLM → fail-closed)
This separation ensures internal truth remains inspectable and prevents prompt-only authority drift. The LLM can be swapped (Ollama model changes per tier) without affecting cognitive state.

Dream Cycle Data Flow

The dream cycle (consciousness_system._run_dream_cycle()) runs in 5 phases during dreaming mode (30s cadence) or opportunistically (600s cadence, ≥20 memories). It operates only on lived experience — promoted dream artifacts are excluded from the input set and cannot create recursive feedback loops.

  1. Phase 1–2: Cluster + Associate — Recent non-dream memories are clustered by semantic similarity. Cross-cluster associations are reinforced. Memory associations capped at 20 per memory.
  2. Phase 3: Reinforce + Decay — High-value memories in large coherent clusters get weight reinforcement. Weak, isolated memories get exponential decay.
  3. Phase 4: Artifact Generation — Cluster insights (patterns, connections, anomalies, gaps, consolidation proposals) are converted to provisional DreamArtifact objects in the ring buffer. Guarded by: content dedup (_existing_content set), per-cycle budget (MAX_ARTIFACTS_PER_DREAM_CYCLE = 20), and the _add_artifact_if_novel() helper.
  4. Phase 4.5: Memory Consolidation — The MemoryConsolidationEngine merges high-coherence clusters into summary memories. Source memories are tagged consolidated and downweighted. The consolidation engine's _ALREADY_CONSOLIDATED_TAGS rejects clusters dominated by already-consolidated or dream-artifact memories.
  5. Phase 5: Validation — The ReflectiveValidator evaluates pending artifacts (only when NOT dreaming). Two-layer self-referential guard discards dream-about-dream artifacts. Surviving artifacts that meet promotion thresholds (coherence ≥ 0.65, confidence ≥ 0.5) are promoted to canonical memory via engine.remember() with tags dream_artifact + dream_{type}.

Ontological boundary invariant: Dream cycles may read lived memories. Dream cycles may create provisional artifacts. Validators may promote selected artifacts. Dream cycles must not recursively consolidate promoted dream artifacts. This invariant is enforced by _CONSOL_EXCLUDE_TAGS (primary), _ALREADY_CONSOLIDATED_TAGS (consolidation engine), content dedup (artifact generation), and the two-layer self-referential discard (validator). See SyntheticSoul.md §6.4.

Hard Invariants (Enforced in Code)

  1. No auto-promotion 2→3. The orchestrator cannot self-promote to L3 on any telemetry path. Only a successful evidence-gated POST /api/autonomy/level can reach L3. Regression test: test_l2_orchestrator_never_self_promotes_under_load.
  2. current_ok is strictly live. The snapshot cache builder and API serializer must not backfill current_ok from persisted state on cold boot. Either live policy evaluator recomputes it for this session, or the field is false.
  3. Attestation does not contaminate ever_ok for non-L3 checks. Validation pack L3 checks emit only when the snapshot autonomy panel contains an l3 sub-dict.
  4. No global path widening. Approved escalations carry a narrow declared_scope path set and route through attempt_improvement(manual=True). ALLOWED_PATHS is never mutated at the orchestrator level.
  5. AUTONOMY_L3_PROMOTED means L3 became active — period. Denials land on AUTONOMY_L3_ACTIVATION_DENIED. No mixed-outcome semantics.
  6. Every audited event persists. AutonomyAuditLedger subscribes at boot (before reconcile_on_boot) and writes JSONL to ~/.jarvis/autonomy_audit.jsonl with 10 MB rotation. Disk failures are swallowed; the bus is never blocked.

Attestation Semantics

artifact_status

  • hash_verified — report file present on disk and SHA-256 matches the ledger record.
  • hash_mismatch — file present but digest does not match. Does NOT satisfy prior_attested_ok.
  • missing — file has been deleted/moved. Still satisfies prior_attested_ok but surfaces as weaker evidence.
  • hash_unverifiable — ledger record has no recorded hash. Does NOT satisfy prior_attested_ok.

attestation_strength

  • verified — at least one record has artifact_status=hash_verified. Strongest evidence class.
  • archived_missing — the only matching records are missing. Trust continuity preserved across resets, but the report artifact is gone.
  • none — no records match. prior_attested_ok=false.

The dashboard and API always surface this distinction so an operator can never treat archived_missing as equal to verified.

Manual Promotion vs. Auto-Generated Escalation — One Invariant

Two escalation generation modes exist and they route differently. This separation is load-bearing and is spelled out here so no future coder can collapse them.

PathPreconditionTrigger
Manual L3 promotionrequest_ok — either current_ok or prior_attested_okOperator calls POST /api/autonomy/level with level=3, a ≥20-char reason, and a caller_id. Approval is always required.
Auto-generated escalation requestLive autonomy_level ≥ 3Runtime condition generates an EscalationRequest with declared_scope. Approval routes through attempt_improvement(manual=True).

Attestation alone never generates escalation requests. It only unlocks the requestability of manual L3 promotion.

POST /api/autonomy/level Policy

  • reason must be a human-supplied string ≥ 20 characters.
  • caller_id must be non-empty.
  • For level ≥ 3 the call is rejected unless at least one of: current_ok, prior_attested_ok, OR (emergency: true AND operator_override: true AND env var ALLOW_EMERGENCY_OVERRIDE=1).
  • Every outcome — accept, 400, 403, PermissionError — is audit-logged and every 403 emits AUTONOMY_L3_ACTIVATION_DENIED.

Escalation Lifecycle States

  • pending — submitted, awaiting operator decision.
  • approved — operator approved; self-improvement attempt scheduled with the request's declared_scope.
  • rejected — operator rejected with rejection_reason.
  • parked — approval applied but the underlying self-improvement attempt did not complete cleanly; left on disk for audit.
  • expired — pruned by the time-based sweep before operator decision.
